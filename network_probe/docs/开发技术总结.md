## 2. 核心技术方案详解

### 2.1 多线程拨测引擎架构

#### 设计思路

拨测引擎是本项目的核心组件。每个引擎（Ping/DNS/Curl/TCP 长连接）都继承自 `QThread`，在独立线程中执行持续性的网络探测任务，通过 Qt 信号-槽机制将结果异步传递给 UI 层，确保界面流畅不卡顿。

#### 引擎基类模式

四个引擎遵循统一的设计模式：

```python
class XxxEngine(QThread):
    result_ready = pyqtSignal(dict)     # 每条探测结果
    error_occurred = pyqtSignal(str)    # 错误消息
    finished_signal = pyqtSignal()      # 拨测完成

    def run(self):
        self._running = True
        while self._running:
            # 1. 执行探测
            # 2. 解析结果 → dict
            # 3. 发射信号 result_ready.emit(record)
            # 4. 精确等待间隔时间
        self.finished_signal.emit()

    def stop(self):
        self._running = False
```

#### 关键技术点

**1. 精确间隔控制**

使用分段轮询方式实现精确的拨测间隔，同时保证停止响应的及时性：

```python
# 使用短间隔轮询以便及时响应停止
end_wait = time.perf_counter() + sleep_time
while self._running and time.perf_counter() < end_wait:
    time.sleep(0.05)  # 50ms 轮询
```

相比简单的 `time.sleep(interval)`，此方案在调用 `stop()` 后最多 50ms 即可终止线程，而非等待完整间隔。

**2. Ping 引擎的跨平台适配**

Ping 引擎通过检测 `platform.system()` 适配 Windows 和 Linux 两种命令格式：

```python
if system == 'windows':
    cmd = ['ping', '-n', '1', '-w', str(timeout_ms), '-l', str(packet_size), ...]
else:
    cmd = ['ping', '-c', '1', '-W', str(timeout_sec), '-s', str(packet_size), ...]
```

使用正则表达式解析 ping 输出，兼容英文和中文 Windows 的输出格式：

```python
rtt_patterns = [
    r'time[=<](\d+\.?\d*)\s*ms',    # 英文 Windows/Linux
    r'时间[=<](\d+\.?\d*)\s*ms',     # 中文 Windows
    r'rtt.*?=\s*[\d.]+/([\d.]+)',    # Linux summary
]
```

**3. DNS 引擎的容错设计**

DNS 引擎对 dnspython 的多种异常进行了细粒度处理：

| 异常类型 | RCODE | 说明 |
|---------|-------|------|
| `dns.resolver.NXDOMAIN` | NXDOMAIN | 域名不存在 |
| `dns.resolver.NoAnswer` | NOERROR | 无应答记录 |
| `dns.resolver.Timeout` | TIMEOUT | 查询超时 |
| `dns.resolver.NoNameservers` | SERVFAIL | 无可用 DNS 服务器 |

同时对 dnspython 做了导入检查，未安装时给出友好提示而非崩溃。

**4. Curl 引擎的阶段耗时拆分**

HTTP 拨测需要展示各阶段耗时（DNS/TCP/TLS/TTFB/传输），由于 requests 库不直接暴露这些细分数据，采用了混合计时方案：

```python
# 独立计时 DNS 解析阶段
dns_start = time.perf_counter()
socket.getaddrinfo(parsed.hostname, port)
dns_time = (time.perf_counter() - dns_start) * 1000

# 总体请求计时
total_start = time.perf_counter()
response = session.request(...)
total_time = (time.perf_counter() - total_start) * 1000

# 近似拆分各阶段（基于比例估算）
tcp_time = min(elapsed * 0.15, total_time * 0.15)
tls_time = min(elapsed * 0.2, total_time * 0.2) if is_https else 0
ttfb_time = elapsed - dns_time - tcp_time - tls_time
transfer_time = total_time - elapsed
```

**5. TCP 长连接引擎的保活探测机制**

TCP 长连接引擎是本项目中最复杂的引擎，采用纯 TCP 保活探测模式（不发送任何应用层数据），通过操作系统级 TCP keepalive 机制实现连接监测。

**工作流程：**

```
建立 TCP 连接（记录建连 RTT）
    → 开启 OS 级 TCP keepalive
    → 循环探活（select + 非阻塞 recv）
    → 连接断开 → 指数退避重连
    → 重复...
```

**连接存活探测原理：**

```python
def _probe_connection(self, sock, timestamp):
    """使用 select() + 非阻塞 recv() 探测 socket 状态"""
    sock.setblocking(False)
    readable, _, exceptional = select.select([sock], [], [sock], timeout_sec)
    sock.setblocking(True)

    if exceptional:
        return {'status': 'error'}          # socket 异常
    if readable:
        data = sock.recv(1024)
        if data:
            return {'status': 'success'}    # 对端发了数据（连接存活）
        else:
            return {'status': 'conn_lost'}  # recv 返回空 = 对端 FIN
    else:
        return {'status': 'success'}        # select 超时 = 连接静默存活
```

**Windows TCP Keepalive 配置：**

```python
# 通过 SIO_KEEPALIVE_VALS ioctl 设置 keepalive 参数
SIO_KEEPALIVE_VALS = 0x98000004
sock.ioctl(SIO_KEEPALIVE_VALS, (1, keepalive_time_ms, keepalive_interval_ms))
```

**指数退避重连策略：**

```python
def _calc_backoff(self) -> float:
    """计算指数退避时间（ms），带 ±20% 随机抖动"""
    base = self._base_backoff_ms * (2 ** min(self._reconnect_attempt, 10))
    capped = min(base, self._max_backoff_ms)  # 最大 30 秒
    jitter = capped * 0.2 * (random.random() * 2 - 1)
    return max(self._base_backoff_ms, capped + jitter)
```

- 基础退避：1 秒
- 最大退避：30 秒
- 退避公式：`min(1s × 2^n, 30s) ± 20% 随机抖动`

**事件类型体系：**

| 事件 | 说明 | UI 图标 |
|------|------|--------|
| `connect` | TCP 建连成功 | 🔗 |
| `connect_fail` | TCP 建连失败（超时/拒绝/错误） | ❌ |
| `heartbeat` | 心跳探活（成功/失败） | 💓 |
| `disconnect` | 连接断开（FIN/RST） | 🔌 |
| `reconnect_wait` | 等待重连（指数退避中） | ⏳ |

def create_record(self, mode, protocol, target, ...):
    existing = self.get_records(mode)
    while len(existing) >= self.MAX_HISTORY:

    # 创建新记录...

### 2.4 统计分析算法

#### 百分位数计算

采用线性插值法计算 P50/P90/P95/P99：

```python
def percentile(sorted_data, p):
    n = len(sorted_data)
    k = (n - 1) * (p / 100.0)
    f, c = math.floor(k), math.ceil(k)
    if f == c:
        return sorted_data[int(k)]
    return sorted_data[int(f)] * (c - k) + sorted_data[int(c)] * (k - f)
```

#### 抖动计算

抖动定义为相邻两次探测 RTT 差值的绝对值：

```python
diffs = [abs(rtt_list[i] - rtt_list[i-1]) for i in range(1, len(rtt_list))]
avg_jitter = sum(diffs) / len(diffs)
max_jitter = max(diffs)
```

#### 最大连续丢包检测

```python
def max_consecutive_loss(statuses):
    max_burst = current_burst = 0
    for s in statuses:
        if s != 'success':
            current_burst += 1
            max_burst = max(max_burst, current_burst)
        else:
            current_burst = 0
    return max_burst
```

#### 质量评级算法

**Ping/DNS/Curl 通用评级：** 三维度独立评分后取最低分作为综合评级，遵循木桶效应：

```
综合评级 = min(丢包率评分, RTT评分, 抖动评分)
```

**TCP 长连接专用评级：** 由于纯 TCP 保活模式不产生心跳 RTT，采用不同的评价维度：

```
综合评级 = min(建连成功率评分, 平均会话时长评分, 建连RTT评分)
```

特殊逻辑：
- 从未断线时，会话时长维度自动评为满分（5分）
- 智能检测服务端固定超时断连行为：多次断线 + 建连全成功 + 会话时长高度一致（变异系数 < 30%），此时会话时长评分提升 1 档，并给出专项说明
- 生成问题诊断说明（如"建连成功率低"、"平均会话时长短"等）

#### TCP 长连接统计指标

```python
def calculate_keepalive_stats(records):
    # 按事件类型分类

                     'libssl-3-x64.dll', 'libcrypto-3-x64.dll']
for _pat in _ssl_dll_patterns:
    if _pat.lower() not in _ssl_already:
        for _sd in _ssl_search_dirs:
            _fp = os.path.join(_sd, _pat)
            if os.path.isfile(_fp):
                a.binaries.append((_pat, _fp, 'BINARY'))
                break
```

### 3.4 SSL 打包问题与解决方案

#### 问题描述

PyInstaller onefile 模式打包后，程序启动时报错：

```
ImportError: DLL load failed while importing _ssl: 内存位置访问无效
```

#### 错误链分析

```
main.py → main_window.py → instant_panel.py → curl_engine.py
    → import ssl → _ssl.pyd → libssl-3.dll + libcrypto-3.dll
    → DLL 加载失败（损坏/缺失）
```

#### 排查过程与经验总结

| 步骤 | 尝试修复 | 结果 |
|------|---------|------|
| 1 | 从 `_EXCLUDE_DIRS` 移除 `'tls'` 目录 | ❌ 未解决 |
| 2 | 在 spec 中添加 OpenSSL DLL 自动查找和强制追加逻辑 | ❌ 未解决 |
| 3 | 添加 `ssl`、`_ssl` 到 `hiddenimports` | ❌ 未解决 |
| 4 | 关闭 `strip=True`（Windows 上 strip 会损坏二进制文件结构） | ❌ 构建成功但仍报错 |
| 5 | 完全禁用 UPX 压缩（`upx=False`） | ❌ 构建成功但仍报错 |
| **6** | **删除 `curl_engine.py` 中无用的 `import ssl`** | **✅ 问题解决** |

#### 根因与最终修复

**直接原因**：`curl_engine.py` 顶部 `import ssl` 在模块加载阶段立即触发 `_ssl.pyd` → `libssl-3.dll` 的加载链。但在 PyInstaller onefile 模式下，UPX 压缩和 strip 处理会损坏这些 DLL 的二进制结构，导致内存映射失败。

**最终修复方案（三管齐下）**：
1. 删除 `curl_engine.py` 中无用的 `import ssl`（该文件通过 `requests` 库间接使用 SSL，无需直接导入）
2. 保持 `strip=False`（防止 Windows 上 `.pyd` 和 `.dll` 被损坏）
3. 保持 `upx=False`（防止 UPX 压缩破坏 DLL 加载）

#### 经验教训

> **⚠️ PyInstaller Windows 打包三条铁律：**
> 1. **永远不要在 Windows 上开启 `strip=True`** —— strip 是 Linux 工具，在 Windows 上会损坏 PE 文件格式
> 2. **谨慎使用 UPX 压缩** —— UPX 压缩 `_ssl.pyd`、`libssl-3.dll` 等关键 DLL 后会导致加载失败
> 3. **避免不必要的 `import`** —— 每个 import 都会在程序启动时触发 DLL 加载链，无用的 import 是定时炸弹

## 4. 代码质量与设计模式

### 4.1 设计模式应用

| 模式 | 应用场景 |
|------|---------|
| 观察者模式 | Qt 信号-槽机制，引擎通过信号通知 UI 更新 |
| 策略模式 | 四种拨测引擎实现相同接口，UI 层按选择实例化 |
| MVC 分层 | UI（View）/ Engine（Controller）/ Storage（Model）清晰分层 |
| 模板方法 | 四个引擎遵循相同的 run/stop 执行框架 |
| FIFO 缓冲 | 历史记录轮换策略 |
| 状态机 | TCP 长连接引擎的连接/探活/断线/重连状态流转 |

### 4.2 错误处理策略

- **引擎层**：所有异常在引擎内部捕获，转换为信号发射，不会导致线程崩溃
  - TCP 长连接引擎额外增加了顶层 `try/except` 兜底，防止任何未预期异常导致应用闪退
- **UI 层**：错误消息以红色文本追加到结果窗口，同时更新状态栏
- **存储层**：文件操作使用 `os.makedirs(exist_ok=True)` 确保目录存在
- **依赖缺失**：DNS 和 HTTP 库使用 `try/except ImportError` 做软依赖检查

### 4.3 线程安全

- 拨测引擎在独立 `QThread` 中运行，通过 Qt 信号队列进行线程间通信
- `_running` 标志位用于优雅停止线程（无需强制终止）
- 窗口关闭时调用 `engine.stop()` + `engine.wait(2000)` 确保线程正确终止
- TCP 长连接引擎的 `_safe_close()` 确保 socket 在任何情况下都能被安全关闭（先 shutdown 再 close）
- CSV 文件追加写入无需加锁（单线程写入）

### 5.6 关键决策与权衡

| 决策点 | 选择 | 理由 |
|--------|------|------|
| GUI 框架 PyQt6 vs tkinter | PyQt6 | 更现代的外观、更丰富的组件、更好的高 DPI 支持 |
| Ping 实现：原生 socket vs 系统命令 | 系统命令 | 原生 ICMP 需要管理员权限，系统命令无此限制 |
| 数据存储：SQLite vs 文件 | 文件（JSON+CSV） | 单机工具无需数据库，文件更轻量、便于查看和导出 |
| TCP 长连接探活：发送数据 vs 纯被动探测 | 纯被动（select+recv） | 不发送应用层数据，避免被服务端因协议不匹配而 RST |
| 重连策略：固定间隔 vs 指数退避 | 指数退避 + 随机抖动 | 避免在服务端异常时产生重连风暴 |
| 历史记录数量限制 | 最多 3 条 | 平衡存储占用与实用性，防止数据无限增长 |
| 打包方式 | 单文件 exe（关闭 UPX 和 strip） | 虽然体积比压缩版大，但稳定性远高于压缩版 |

### 5.7 已知限制

1. **Ping 引擎需要系统权限**：某些 Windows 防火墙策略可能阻止 ICMP
2. **Curl 各阶段耗时为近似值**：requests 库不直接暴露 TCP/TLS 分段耗时，采用比例估算
3. **TCP 长连接不产生心跳 RTT**：纯被动探测模式下，探活结果为"存活/断开"二值，无法测量精确的心跳延迟
4. **无图表可视化**：当前版本仅提供文本和数值统计，未集成 RTT 趋势图等图表
5. **单目标拨测**：每次只能拨测一个目标地址
