# 企业IM安卓客户端包信息爬虫 — 开发需求文档

## 一、项目概述

开发一个自动化爬虫程序，通过**三级递进式搜索策略**，全网爬取 **企业微信、钉钉、飞书** 及其衍生定制客户端的安卓应用包（APK）公开信息。

**核心思路**：
1. **第一级**：从产品官网和公开案例页挖掘客户企业名称清单
2. **第二级**：以客户名称为线索，全网搜索其对应的定制安卓客户端包
3. **第三级**：从主流应用商店和第三方平台中检索补全包信息

## 二、目标产品范围

| 产品线 | 官方包名（主包） | 定制包识别策略 |
|--------|-----------------|---------------|
| 企业微信 | `com.tencent.wework` | 包名含 `wework`/`wecom`、应用名含"企业微信"等关键词 |
| 钉钉 | `com.alibaba.android.rimet` | 包名含 `dingtalk`/`rimet`、应用名含"钉钉"等关键词 |
| 飞书 | `com.ss.android.lark` | 包名含 `lark`/`feishu`、应用名含"飞书"等关键词 |

> **定制包**：各企业基于上述产品二次打包或由厂商为特定客户发布的专属版本（如"XX集团企业微信"、"XX政务钉钉"等）。

## 三、爬取字段

每条记录须包含以下字段：

| 字段 | 说明 | 示例 |
|------|------|------|
| `package_name` | 安卓包名（唯一标识） | `com.tencent.wework` |
| `app_name` | 应用显示名称 | `企业微信` |
| `enterprise_name` | 使用/定制该包的企业名称 | `XX集团` |
| `product_line` | 所属产品线 | `企业微信/钉钉/飞书` |
| `source_site` | 来源网站域名 | `app.mi.com` |
| `source_url` | 详情页完整URL | `https://app.mi.com/details?id=xxx` |
| `version` | 版本号 | `4.1.16` |
| `version_code` | 版本代码（如可获取） | `4011600` |
| `update_date` | 最近更新日期 | `2026-01-15` |
| `download_count` | 下载量（如可获取） | `1000万+` |
| `description` | 应用简介（截取前500字） | `...` |
| `developer` | 开发者/发布者名称 | `腾讯科技` |
| `crawl_time` | 爬取时间戳 | `2026-02-09 10:30:00` |
| `discovery_method` | 发现方式 | `官网案例/搜索引擎/应用商店` |

## 四、三级递进式搜索策略

### ▎整体流程

```
┌─────────────────────────────────────────────────────────────────┐
│                     第一级：客户名称挖掘                          │
│                                                                 │
│  产品官网案例页 → 合作伙伴页 → 新闻报道 → 行业白皮书              │
│          ↓                                                      │
│   输出：企业客户名称清单（预估 1000~5000+ 家）                    │
└──────────────────────────┬──────────────────────────────────────┘
                           ↓
┌──────────────────────────▼──────────────────────────────────────┐
│                 第二级：全网搜索定制包                             │
│                                                                 │
│  搜索引擎全网搜索："{企业名} + 企业微信/钉钉/飞书 + 安卓/App"     │
│  扫描结果：企业官网下载页、政务平台、行业门户、APK收录站            │
│          ↓                                                      │
│   输出：定制包候选列表（名称 + 下载链接/商店链接）                 │
└──────────────────────────┬──────────────────────────────────────┘
                           ↓

```

### 4.2 第二级：全网搜索定制包

以第一级挖掘到的企业名称为输入，全网搜索该企业是否有对应的定制安卓客户端。

#### 4.2.1 搜索策略

针对每个客户企业名称，组合搜索：

```
搜索模板：
  "{企业名}" "企业微信" App 下载 安卓
  "{企业名}" "钉钉" 定制 客户端 Android
  "{企业名}" "飞书" 专属版 apk
  "{企业名}" 移动办公 App 安卓 下载
  "{企业名}" OA App 下载
```

#### 4.2.2 搜索引擎覆盖

| 搜索引擎 | 说明 | 优先级 |
|----------|------|--------|
| 百度搜索 | 中文内容覆盖最广 | P0 |
| 必应搜索 | 补充百度遗漏 | P1 |
| 搜狗搜索 | 微信公众号内容独家 | P1 |
| 360搜索 | 部分独家收录 | P2 |
| 神马搜索 | 移动端内容 | P2 |

#### 4.2.3 搜索结果页分析

对搜索结果中的每个链接进行分析：

| 结果类型 | 处理方式 |
|----------|---------|
| 应用商店详情页 | 直接提取包信息 → 进入第三级补全 |
| 企业官网下载页 | 提取App名称和下载链接，解析APK文件名推断包名 |
| 政府门户公告页 | 提取App名称、使用说明中的包名或下载链接 |
| 新闻/公告文章 | 提取App名称，再二次搜索获取包名 |
| APK收录站 | 直接提取包名和元数据 |

#### 4.2.4 企业官网深度扫描

对搜索到的企业官网，自动扫描以下路径寻找App下载入口：

```
常见路径模式：
  /download, /app, /mobile, /android
  /about/app, /service/download
  页面内搜索关键词：安卓下载, Android, App下载, 扫码下载, 移动客户端
```

## 五、搜索与匹配策略

### 5.1 包名匹配规则

```python
PACKAGE_PATTERNS = {
    '企业微信': [
        r'com\.tencent\.wework.*',
        r'.*wework.*',
        r'.*wecom.*',
    ],
    '钉钉': [
        r'com\.alibaba\.android\.rimet.*',
        r'.*dingtalk.*',
        r'.*rimet.*',
    ],
    '飞书': [
        r'com\.ss\.android\.lark.*',
        r'.*lark.*',
        r'.*feishu.*',
    ]
}
```

### 5.2 应用名关键词匹配

```python
APP_NAME_KEYWORDS = {
    '企业微信': ['企业微信', '企微', 'WeWork', 'WeCom', '政务微信'],
    '钉钉':    ['钉钉', 'DingTalk', '政务钉', '浙政钉', '粤政易',
                '赣政通', '鄂汇办', '苏政通', '豫政通', '鲁政通',
                '闽政通', '皖政通'],
    '飞书':    ['飞书', 'Lark', 'Feishu', '飞书办公'],
}
```

### 5.3 企业名称提取策略

| 提取方法 | 说明 | 示例 |
|----------|------|------|
| 应用名解析 | 去掉产品关键词后的前缀/后缀 | "XX集团企业微信" → "XX集团" |
| 开发者字段 | 应用商店的发布者/开发者信息 | 开发者: XX科技有限公司 |
| 应用描述NLP | 从描述文本中抽取组织实体 | "为XX银行量身打造…" → "XX银行" |
| 第一级客户库反查 | 用已有客户名匹配 | 客户库中已有该企业 |
| 搜索结果上下文 | 从搜索结果摘要中提取 | "XX市政府上线政务钉钉…" |

### 5.4 定制包判定规则

一条记录被判定为目标定制包需满足以下条件之一：

1. 包名命中 `PACKAGE_PATTERNS` 中的正则规则
2. 应用名命中 `APP_NAME_KEYWORDS` 中的关键词
3. 应用描述中同时出现 **产品关键词 + 企业/定制/专属** 等词汇
4. 第一级客户名称 + 搜索引擎结果交叉验证命中

## 七、反爬与合规策略

### 7.1 反爬措施

| 策略 | 实现方式 |
|------|---------|
| 请求限速 | 每个站点间隔 3~8 秒随机延迟；搜索引擎间隔 10~15 秒 |
| UA轮换 | 维护 50+ 常见浏览器 User-Agent |
| 代理IP | 接入付费代理池，自动轮换 |
| Cookie管理 | 模拟正常浏览器Cookie |
| JS渲染 | Playwright无头浏览器处理动态加载页面 |
| 验证码 | 对接打码平台（仅必要时） |
| 重试机制 | 指数退避重试，最多3次 |
| 搜索引擎限速 | 单引擎每日限制请求数，多引擎轮换 |

### 7.2 合规要求

- **仅爬取公开可访问页面**，不绕过登录墙或付费墙
- **遵守 robots.txt 规则**
- **不下载 APK 文件本身**，仅采集元数据信息
- **控制爬取频率**，避免对目标站点造成负担
- 爬取数据**仅用于内部分析**，不进行公开传播
- **搜索引擎使用**遵守各引擎服务条款
